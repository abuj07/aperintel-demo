Source-System Permission Mirroring
Permissions from SharePoint, OneDrive, Teams and Google Drive are captured at ingestion time and stored on every document chunk. Users can only retrieve documents they're already authorised to see in the source — enforced at database query level, not application logic.

How Permission Mirroring Works
1
Permissions fetched at ingestion
When a document is ingested, the connector calls the source system's permissions API (e.g. Microsoft Graph /permissions endpoint) and retrieves the complete list of users and groups authorised to access that document.
2
Scope stored on every chunk
Every chunk produced from a document is tagged with a permission_scope array containing authorised user IDs and group IDs. This is stored alongside the vector embedding in pgvector.
3
Hard pre-filter at query time
Before any vector similarity search occurs, the retrieval pipeline filters the index to only chunks where the requesting user's ID or group memberships intersect with permission_scope. Applied at database level — no bypass path exists.
4
Delta sync every 15 minutes
Permission changes in the source are automatically reflected. When a user is removed from a SharePoint group, the next delta sync updates permission_scope on all affected chunks with no manual intervention.
